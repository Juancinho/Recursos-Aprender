# BM25

BM25 es un método de recuperación sparse/léxica. No entiende semántica profunda; puntúa documentos según coincidencias de términos, rareza de términos y longitud del documento.

## Cuándo funciona muy bien

- Siglas: UDS, DoIP, ECU, NRC.
- IDs: `0x22`, `0x31`.
- Nombres exactos de funciones.
- Errores/logs literales.

## Cuándo falla

- Parafraseo fuerte.
- Sinónimos.
- Relaciones implícitas.

## Comparación con embeddings

BM25 puede ganar cuando el token exacto es crítico. Embeddings pueden ganar cuando el texto usa palabras distintas para la misma idea. [[Hybrid_Search]] combina ambos.

## Fórmula mental

Más peso a términos raros que aparecen en documentos no demasiado largos. No necesitas memorizar la fórmula para usarlo bien, pero sí entender que es lexical.

## Ampliación curso: por qué BM25 sigue siendo muy fuerte

BM25 no "entiende" UDS, pero si la query contiene `0x22` y el documento contiene `0x22`, esa coincidencia exacta puede ser más valiosa que una similitud semántica suave.

### Componentes de la intuición

- **Term frequency**: si un término aparece, importa; si aparece demasiadas veces, se satura.
- **Inverse document frequency**: términos raros pesan más.
- **Length normalization**: documentos muy largos no deberían ganar solo por contener más palabras.

### Caso donde BM25 gana

Query: "NRC 0x31 unsupported DID"

Un embedding puede recuperar "diagnostic error handling" porque semánticamente suena parecido. BM25 recuperará documentos con `NRC`, `0x31`, `DID`, que probablemente son los exactos.

### Caso donde BM25 pierde

Query: "data identifier read service"

Documento: "UDS service 0x22 ReadDataByIdentifier"

Si no comparte términos suficientes, BM25 puede no verlo. Ahí embeddings ayudan.
